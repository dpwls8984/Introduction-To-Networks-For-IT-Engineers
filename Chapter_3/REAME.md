# Chapter.3 네트워크 통신하기

### 3.1 네트워크에서 출발지에서 목적지로 데이터를 전송할 때 사용하는 통신 방식 종류

1. 유니캐스트
    1. 1:1 통신
    2. 출발지와 목적지가 1:1로 통신
    3. 실제 사용하는 대부분 통신은 유니캐스트 방식을 사용
2. 브로드캐스트
    1. 1:모든 통신
    2. _동일 네트워크에 존재하는 모든 호스트가 목적지_
        1. *허브같은 느낌인가? 1계층에서 2계층으로 올라갈 때 전기신호를 모든 port에 다 쏜다고 했는데 그 과정과 동일한 통신 방식인가?*
    3. 목적지가 ‘모든’으로 표기되어있는 통신 방식
    4. 유니캐스트로 통신하기 전, 주로 상대방의 정확한 위치를 알기 위해 사용된다
3. 멀티캐스트
    1. 1:그룹 (멀티캐스트 구독 호스트) 통신
    2. 하나의 출발지에서 다수의 특정 목적지로 데이터 전송
    3. **멀티캐스트 그룹 주소**를 이용해 해당 그룹에 속한 다수의 호스트로 패킷을 전송
    4. ex) 사내 방송, 증권 시세 전송같이 단방향으로 다수에게 동시에 같은 내용 전달할 때
4. 애니캐스트
    1. 1:1 통신(목적지는 동일 그룹 내의 1개 호스트)
    2. 다수의 동일 그룹 중 가장 가까운 호스트에서 응답
    3. _IPv4에서는 일부 기능 구현, IPv6은 모두 구현 가능_
        1. _*기능 구현한다는 게 무슨 뜻인지?*_
    4. 애니캐스트 주소가 같은 호스트들 중에서 **가장 가깝거나 가장 효율적으로** 서비스 할 수 있는 호스트와 통신하는 방식
        1. 가장 가까운 DNS 서버를 찾을 때 사용하거나 가장 가까운 게이트웨이를 찾는 애니캐스트 게이트웨이 기능에 사용하기도 함

> **유니캐스트와 애니캐스트의 차이점은 무엇인가?**
> 

통신할 수 있는 후보자가 서로 다르다.

- 유니캐스트: 출발지와 목적지가 모두 한 대씩
- 애니캐스트: 같은 목적지 주소를 가진 서버가 여러 대여서 통신 가능한 다수의 후보군이 있다

| 타입 | 통신 대상 | 범위 | IPv4 | IPv6 | 예제 |
| --- | --- | --- | --- | --- | --- |
| 유니캐스트 | 1:1 | 전체 네트워크 | O | O | HTTP |
| 브로드캐스트 | 1:모든 | 서브넷(로컬 네트워크) | O | X | ARP |
| 멀티캐스트 | 1:그룹 | 정의된 구간 | O | O | 방송 |
| 애니캐스트 | 1:1 | 전체 네트워크 | △ | O | 6 to 4 <br>DNS |

⇒ 통신 방식을 구분할 때 중요한 점은 실제 데이터를 전달하려는 출발지 기준이 아닌, **목적지 기준으로 구분한다**

### 3.2 MAC 주소


```yml
- Media Access Control
- 2계층(데이터 링크 계층)에서 통신을 위해 네트워크 인터페이스에 할당된 고유 식별자
- 네트워크에 접속하는 모든 장비는 MAC 주소라는 물리적인 주소가 있어야 함.
```

- MAC의 주소 체계
    - 한 제조업체에 하나 이상의 주소풀을 주고 그 풀 안에서 각 제조업체가 자체적으로 MAC 주소 할당
    - **제조사 코드(Vendor Code)**: 한 제조업체에 하나 이상의 주소풀을 주는 것
        - 이 주소는 국제기구인 IEEE가 관리
    - 48비트의 16진수 12자리로 표현
        - 제조사 코드가 앞의 24비트 담당(OUI),
        - 각 제조사에서 자체적으로 할당한 뒤의 24비트(UAA)로 나뉨
    - 하드웨어적으로 정해져 나오므로 MAC = BIA (Burned-In Address)
- MAC 주소 동작
    - NIC가 자신의 MAC 주소를 갖고있고, 전기신호가 들어오면 2계층에서 데이터 형태(패킷)로 변환하여 내용을 구분한 뒤 도착지 MAC 주소를 확인한다.
    - 만약 도착지 MAC 주소가 자신의 MAC 주소와 다르면 그 패킷을 폐기
    - 패킷의 목적지가 내 주소이거나 브로드캐스트(1: 모든 통신), 멀티캐스트(1: 그룹)와 같은 그룹 주소면 처리해야 할 주소로 인지해 패킷 정보를 상위 계층으로 넘겨준다
        - _본인의 주소, 브로드캐스트(1: 모든 통신) 주소는 NIC 자체적으로 패킷을 처리하는 게 아니라 OS나 애플리케이션에서 처리해야 하므로 시스템에 부하가 작용한다._
            - _*4.3.1 루프란? 절에 나오는 브로드캐스트 스톰 내용 배울 때 다시 한 번 알아보기*_
            - 브로드캐스트 스톰의 경우, 브로드캐스트가 회선을 모두 채우게 되는데 네트워크에 연결된 모든 단말이 브로드캐스트를 처리하느라 CPU 사용량이 증가한다

### 3.3 IP 주소


```yml
- 사용자가 변경 가능한 논리 주소이다
- 주소에 레벨이 있다. 그룹을 의미하는 네트워크 주소와 호스트 주소로 나뉜다.
```

- IP 주소 체계
    - 흔히 사용하는 IP 주소는 32비트인 IPv4 주소이고, IPv6 주소는 128 비트이다.
    - IPv4 주소를 표기할 때 4개의 옥텟(Octet)이라고 불리는 8비트 단위로 나누고 각 옥텟은 “.”으로 구분한다.
        - 2계층의 MAC 주소가 16진수로 표기된 것과 달리 IP 주소는 10진수로 표기하므로 8비트 옥텟은 0~255의 값을 쓸 수 있다.
    - IP주소는 네트워크 주소와 호스트 주소 두 부분으로 나뉘지만, **구분하는 경계점이 고정되어있지 않다.**
        - 네트워크 주소
            - 호스트들을 모든 네트워크를 지칭하는 주소
            - 네트워크 주소가 동일한 네트워크를 로컬 네트워크라고 함
        - 호스트 주소
            - 하나의 네트워크 내에 존재하는 호스트들을 구분하기 위한 주소
    - IP 주소 체계는 필요한 호스트 IP 개수에 따라 네트워크의 크기를 다르게 할당할 수 있는 **클래스(Class) 개념**을 도입했다.
        - 다른 고정된 네트워크 체계에 비해 주소를 절약할 수 있다는 장점 있음.
          <img width="2048" height="1423" alt="image" src="https://github.com/user-attachments/assets/e2489db4-30a8-42fe-8310-ec6208020d79" />
        
        - A 클래스가 가장 큰 주소를 갖는다. (1,600만 개의 IP 주소)
        - B 클래스: 약 6만 5천개
        - C 클래스: 약 250개
        - 네트워크 주소와 호스트 주소를 나누는 구분자가 고정되어있지 않으므로 네트워크 크기가 달라질 수 있다.
            - 클래스 개념이 없었다면 네트워크 크기가 모두 같아지고, 큰 네트워크를 필요로 하는 조직은 네트워크를 여러 개 확보해야하는 어려움이 있고 연속된 네트워크를 할당받기 어렵다.
            - 작은 네트워크가 필요한 조직의 입장에서는 너무 많은 IP를 가져가므로 IP가 낭비된다.
              
              <img width="764" height="183" alt="image" src="https://github.com/user-attachments/assets/eef0e67a-9ab4-4ca9-9fe9-eab01f383c0e" />
- 하지만 _현재는 클래스 기반으로 네트워크를 분할하지 않고, 더 세밀하게 분할하고 할당하기 위해 필요한 네트워크 크기에 맞춰 1비트 단위로 네트워크를 상세히 분할하는 방법을 사용한다._
    - _*지금 사용하고 있지 않다면 왜 클래스 기반 네트워크 분할법을 배워야하는가?*_
        
        클래스 기반 방식(Classful)은 주소의 첫 번째 덩어리만 보고도 "아, 이건 C클래스구나!"라고 바로 알 수 있었습니다. 하지만 이 방식은 낭비가 너무 심했죠. 100명만 사는 회사에 6만 개짜리 B클래스 주소를 줘야 했으니까요.
        
        이를 해결하기 위해 등장한 것이 **서브넷 마스크**입니다. 클래스라는 고정된 틀을 깨고 주소를 쪼개기 시작한 것이죠. 하지만 면접이나 자격증 시험에서 클래스를 묻는 이유는, 여러분이 "네트워크 주소와 호스트 주소를 구분할 줄 아는가?"라는 근본적인 능력을 갖췄는지 확인하기 위함입니다.
        
        ---
        
        백엔드 개발자가 클래스 개념을 반드시 알아야 하는 실무적인 이유는 **사설 IP 대역(Private IP)** 때문입니다. 여러분이 AWS VPC(Virtual Private Cloud)를 설정하거나 회사 내부 서버 망을 구축할 때, 아무 숫자나 쓰는 게 아닙니다.
        
        - **10.0.0.0/8 (A클래스 대역)**
        - **172.16.0.0/12 (B클래스 대역 일부)**
        - **192.168.0.0/16 (C클래스 대역)**
        
        전 세계적으로 약속된 사설 IP 대역은 여전히 클래스 구분을 기반으로 할당되어 있습니다. 만약 여러분이 대규모 마이크로서비스 아키텍처(MSA)를 설계한다면, 수만 개의 컨테이너가 소통할 수 있도록 A클래스 대역(10.x.x.x)을 선택해야 하는 판단력을 가져야 합니다.
        
        ---
        
        실무 환경은 항상 최신 기술로만 가득 차 있지 않습니다. 금융권이나 오래된 공공기관의 인프라에는 여전히 클래스 기반으로 동작하는 오래된 라우터나 장비(Legacy)가 남아있을 수 있습니다.
        
        특히 **RIP(Routing Information Protocol) v1** 같은 오래된 라우팅 프로토콜은 서브넷 마스크 정보를 전달하지 않고 클래스 단위로만 길을 찾습니다. 이런 환경에서 클래스 개념을 모른 채 CIDR 방식으로 네트워크를 쪼개버리면, 특정 구간에서 통신이 끊기는 대참사가 발생합니다. 전문가는 과거와 현재를 잇는 '하위 호환성'을 고려해야 합니다.
        
        ---
        
        백엔드 개발자로서 클래스 개념과 그 변천사를 알아야 하는 최종적인 이유는 **IP 주소 고갈 문제와 오버레이 네트워크(Overlay Network)** 이해에 있습니다.
        
        현대 개발자는 쿠버네티스(Kubernetes)를 사용하죠. 쿠버네티스 내부의 Pod들은 각각 고유한 IP를 가집니다. 이때 클러스터 네트워크를 설정하면서 "왜 주소 대역을 이렇게 잡았지?"라는 의문에 답하려면, 클래스 기반의 할당 원칙과 이를 극복하기 위한 CIDR의 관계를 명확히 파악하고 있어야 합니다. 주소 체계의 '근본'을 모르면 복잡한 트러블슈팅 상황에서 네트워크 계층(L3)의 문제를 애플리케이션 계층(L7)의 문제로 오해하여 시간을 허비하게 됩니다.
        

- 기존 IP 주소 체계(Classful)의 문제점
    - 상위 클래스 (A class)에서 할당받은 조직에서 이 주소들을 제대로 사용하지 못하면서 낭비하는 것
        - Classful에서는 한 개의 클래스 네트워크가 한 조직에 할당되면 아무리 비어있는 주소라도 IP를 분할해 다른 기관이 사용할 수 없도록 한다.
    - IP 주소 부족과 낭비 문제를 해결하기 위해 만들어낸 3가지 전략
        - CIDR (Classless Inter-Domain Routing)
            - 현재 우리가 사용하는 주소 체계로, 클래스 개념을 적용하지 않는다
            - 기존엔 앞 숫자를 보고 클래스 확인한 후 어디까지가 네트워크 주소이고 호스트 주소인지 구분할 수 있었지만,
            - **Classless network에서는 별도로 네트워크와 호스트 주소를 나누는 구분자**를 사용해야한다. 이를 _**서브넷 마스크(Subnet Mask)**_라고 부른다.
                - 서브넷 마스크는 IP 주소와 네트워크 주소를 구분할 때 사용하고, 2진수 숫자 1은 네트워크 주소, 0은 호스트 주소로 표시한다.
            - 새로운 네트워크-호스트 구분 기준을 사용자가 정해 원래 클래스풀 단위의 네트워크보다 더 쪼개 사용하는 것을 **서브네팅**이라 한다.
                - 실제로는 옥텟 단위보다 더 잘게 쪼개 2진수의 1비트 단위로 네트워크를 분할한다.
            - 네트워크 사용자의 서브네팅 (이미 분할된 네트워크에서 자신의 네트워크와 원격지 네트워크를 구분해야할 때)
                - 사용할 수 있는 IP 주소 범위를 파악해야한다.
                - 서브넷 마스크를 통해 이를 중심으로 계산을 하면 된다
            - 네트워크 설계자의 서브네팅 (네트워크를 효율적으로 어떻게 분할할 것인지 계획할 때)
                - 서브넷 된 하나의 네트워크에 IP를 몇 개나 할당해야하는가 (PC는 몇 대 있는가)
                - 그리고 서브넷 된 네트워크가 몇 개나 필요한가
        - NAT와 사설 IP 주소
            - NAT(Network Address Translation): 라우터나 방화벽 장비에서 사설 IP 주소(Local IP)를 공인 IP 주소(Public IP)로 변환하여, 내부 네트워크의 여러 기기가 하나의 공인 IP로 인터넷에 접속할 수 있게 하는 기술
                - ex) 공유기
            - 사설 IP 주소: 인터넷에 연결하지 않고 개인적으로 네트워크를 구성할 때 사용하는 IP 주소
                - 사설 IP를 이용하면 인터넷에 직접 접속은 못하지만 NAT 장비에서 공인 IP로 변경한 후 인터넷 접속이 가능하다.
            - 회사 내부에서 사설 네트워크 구축할 때 NAT을 사용해서 인터넷에 연결하더라도 다른 사용자에게 할당된 IP를 사설 네트워크 주소로 사용하면 안된다. 인터넷 어느 구간에서도 사용되지 않는 RFC에 명시된 사설 IP 대역을 사용할 것을 추천한다.
        - IPv6

### 3.4 TCP와 UDP

> 4계층 프로토콜의 목적: 
목적지 단말 안에서 동작하는 여러 애플리케이션 프로세스 중 **통신해야 할 목적지 프로세스를 정확히 찾아가고 패킷 순서가 바뀌지 않도록 원래 데이터를 잘 조합해 만들어내기 위한 역할**
> 


```yml
- TCP: 연결 지향형, 데이터 흐름 제어, 에러 제어
    - 인터넷을 통해 직접 연결되고 나서야 데이터를 주고받을 수 있는 관계
- UDP: 많은 데이터, 신뢰성 보장 없음, 단순
    - 연결 없음, 데이터 그냥 보냄. (IP 주소는 당연히 있음. 그러면 그 주소를 따라 알아서 보내짐)
```

- TCP/IP 4계층에서 통신해야 할 목적지 프로세스를 정확히 찾아가고 패킷 순서가 바뀌지 않도록 원래 데이터를 잘 조합해 만들어내기 위한 역할을 TCP와 UDP가 담당한다.
    - 패킷을 분할하고 조합하기 위해 TCP 프로토콜에서는 시퀀스 번호와 ACK 번호를 사용한다
    - _TCP/IP 프로토콜 스택_에서 4계층의 상위 프로토콜 지시자는 포트 번호이며, 출발지와 목적지를 구분해 처리해야 한다.
        - _TCP/IP 프로토콜 스택과 TCP/IP 4계층은 같은 말인가?_
- TCP
    - TCP에서는 분할된 패킷을 잘 분할하고 수신 측이 잘 조합하도록 패킷에 순서를 주고 응답 번호를 부여한다
        - 패킷에 순서 부여: 시퀀스 번호
        - 응답 번호 부여: ACK 번호
    - 보내는 쪽에서 패킷에 번호를 부여하고 받는 쪽은 이 번호의 순서가 맞는지 확인한다. 받은 패킷 번호가 맞으면 응답을 주는데 이때 다음 번호의 패킷을 요청한다. 이 숫자를 ACK 번호라고 한다.
    - 즉, 일방적으로 패킷을 보내는 것이 아니라 상대방이 잘 받았는지 확인하기 위해 ACK 번호를 확인하고 다음 패킷을 전송한다.
    - 하지만 한 패킷을 보낼 때마다 응답 번호를 받고 다시 데이터를 전송한다면 모든 데이터를 전송하는 데 긴 시간이 걸릴 것이므로, 데이터를 보낼 때 많은 패킷을 한꺼번에 보내고 응답을 하나만 받는다.
        - 이때 적절한 송신량을 경정해야하는데 한 번에 받을 수 있는 데이터 크기를 **윈도 사이즈**라 하고 네트워크 상황에 따라 윈도 사이즈를 조절하는 것을 **슬라이딩 윈도**라고 한다.
    - 3-way handshake
        - TCP에서는 통신 시작 전 사전 연결 작업을 진행하는데, 3번의 패킷을 주고 받으면서 통신을 서로 준비한다.
        
        | **단계** | **동작 (Packet)** | **클라이언트 상태** | **서버 상태** |
        | --- | --- | --- | --- |
        | **준비** | - | `CLOSED` | **`LISTEN`** |
        | **1단계** | **SYN** 전송 | **`SYN_SENT`** | `LISTEN` |
        | **2단계** | **SYN+ACK** 수신/전송 | `SYN_SENT` | **`SYN_RECEIVED`** |
        | **3단계** | **ACK** 전송/수신 | **`ESTABLISHED`** | **`ESTABLISHED`** |
        - TCP flag
        
        | **플래그 (Flag)** | **이름 (Full Name)** | **의미와 역할** |
        | --- | --- | --- |
        | **SYN** | Synchronize | **연결 요청.** 시퀀스 번호를 동기화하여 대화를 시작함. |
        | **ACK** | Acknowledgment | **응답 확인.** 패킷을 잘 받았음을 상대에게 알림. |
        | **FIN** | Finish | **연결 종료.** 데이터 전송이 끝났으니 정상적으로 끊자고 함. |
        | **RST** | Reset | **강제 종료.** 연결에 문제가 생겨 즉시 연결을 초기화함. |
        | **PSH** | Push | **밀어넣기.** 버퍼가 찰 때까지 기다리지 말고 즉시 앱으로 전달함. |
        | **URG** | Urgent | **긴급 데이터.** 우선순위가 높은 데이터가 포함되어 있음을 알림. |
- UDP
    - 데이터 전송을 보장하지 않는 프로토콜이므로 제한된 용도로만 사용된다
    - 단방향으로 다수의 단말과 통신해 응답을 받기 어려운 환경에서 주로 사용된다
        - 데이터를 전송하는 데 신뢰성보다 일부 데이터가 유실되더라도 시간에 맞추어 계속 전송하는 것이 중요한 서비스에서 UDP를 사용한다. ex) 화상 연결
    - TCP와 달리 통신 시작 전 사전 연결 점검 절차가 없다. 그 대신 UDP에서 첫 데이터는 리소스 확보를 위해 인터럽트를 거는 용도로 사용되고 유실된다.

### 3.5 ARP

<aside>

IP 주소 체계는 MAC 주소와 연관성이 없으므로 두 개의 주소를 연계시켜 주기 위한 매커니즘이 필요한데, 이때 사용되는 포로토콜이 ARP이다.

상대방의 MAC 주소를 알아내기 위해 사용되는 프로토콜이 ARP(Address Resolution Protocol)이다.

</aside>

- ARP
    - ARP는 논리적 주소(IP 주소)를 기반으로 물리적 주소(MAC 주소)를 알아내기 위해 사용하는 프로토콜입니다.
    - 네트워크 계층(L3)에서는 IP 주소를 사용하여 목적지를 식별하지만, 실제 데이터 링크 계층(L2)에서 데이터를 전송하기 위해서는 LAN 카드에 할당된 고유한 번호인 MAC 주소가 반드시 필요합니다. ARP는 이 두 주소 사이의 다리 역할을 합니다.
- ARP의 동작 과정
    1. **ARP 요청 (Broadcasting):** 송신자는 목적지 IP는 알지만 MAC 주소를 모를 때, 네트워크 상의 모든 PC에게 "IP가 `192.168.0.5`인 사람, MAC 주소 좀 알려줘!"라는 메시지를 보냅니다.
    2. **ARP 응답 (Unicast):** 해당 IP를 가진 호스트만이 자신의 MAC 주소를 담아 송신자에게 응답을 보냅니다.
    3. **ARP 캐싱:** 송신자는 받은 MAC 주소를 **ARP Table**에 저장하여, 다음 통신부터는 매번 묻지 않고 바로 데이터를 보낼 수 있게 합니다.

- GARP
    
    **GARP**는 송신 IP와 타겟 IP를 **자신의 IP 주소로 설정**하여 전송하는 특수한 형태의 ARP입니다. 즉, 누구의 주소를 묻는 것이 아니라 자신의 정보를 네트워크에 알리는 용도입니다.
    
    - **IP 중복 체크:** 내가 사용하려는 IP를 다른 사람이 이미 쓰고 있는지 확인합니다.
    - **ARP 테이블 갱신:** 자신의 MAC 주소가 바뀌었을 때(예: 서버 이중화 구성에서 Master가 바뀐 경우) 주변 장치들의 ARP 테이블을 최신화하라고 알립니다.
- RARP
    
    **RARP**는 ARP와 반대로, 물리적 주소(MAC 주소)를 가지고 논리적 주소(IP 주소)를 서버로부터 할당받기 위해 사용했던 프로토콜입니다.
    
    - **과거의 용도:** IP 설정 정보가 없는 하드디스크가 없는 단말기(Diskless 워크스테이션)가 부팅될 때 자신의 IP를 서버에 물어보는 용도로 사용되었습니다.
    - **현재:** 지금은 기능이 더 확장된 DHCP(Dynamic Host Configuration Protocol)가 이 역할을 완전히 대체했기 때문에 거의 사용되지 않습니다.
    
    | **구분** | **목적** | **핵심 매커니즘** |
    | --- | --- | --- |
    | **ARP** | IP → MAC | 주소 식별 (통신 시작) |
    | **GARP** | 내 IP → 내 MAC | 중복 체크 및 테이블 갱신 |
    | **RARP** | MAC → IP | 초기 IP 할당 (현재는 DHCP 사용) |

### 3.6 서브넷과 게이트웨이

- 서브넷과 게이트웨이의 용도는 무엇인가
    - 서브넷
        - 용도: 브로드캐스트 패킷(전체 알림)이 네트워크 전체로 퍼지는 범위를 제한하여 네트워크 성능을 높이고, 보안상 네트워크를 격리하기 위해 사용합니다.
    - 게이트웨이
        - 용도: 내 서브넷 내부에 없는 목적지(예: 외부 인터넷이나 다른 부서 네트워크)로 패킷을 보낼 때, 일단 모든 패킷을 게이트웨이(일반적으로 라우터)로 던집니다.
- 프로시 ARP는 무엇인가
    - 원래 ARP는 동일한 서브넷 안에서만 동작합니다. 하지만 프록시 ARP는 호스트가 다른 네트워크에 있는 목적지의 MAC 주소를 물어볼 때, 라우터가 자기 자신의 MAC 주소를 대신 응답해주는 기술입니다.
    - 용도: 호스트에 게이트웨이 설정이 되어 있지 않거나, 네트워크 구조가 복잡하여 투명한 연결이 필요할 때 사용합니다.
- 2계층 통신과 3계층 통신은 무슨 관계인가
    - 네트워크 통신은 항상 3계층(IP) 정보로 경로를 찾고, 2계층(MAC) 정보로 실제 데이터를 실어 나르는 구조입니다.
    
    | **구분** | **단위** | **주소 체계** | **역할** |
    | --- | --- | --- | --- |
    | **3계층 (Network)** | 패킷 (Packet) | IP 주소 | **최종 목적지**를 찾아가는 지도 역할 (Routing) |
    | **2계층 (Data Link)** | 프레임 (Frame) | MAC 주소 | **인접한 장비**끼리 데이터를 전달하는 실제 발 역할 (Forwarding) |
- **캡슐화와 역캡슐화**
    1. **3계층의 결정:** 데이터가 나갈 때, 먼저 3계층에서 "목적지 IP가 내 네트워크인가?"를 확인합니다.
    2. **2계층의 실행:** 같은 네트워크라면 목적지의 MAC 주소를, 다른 네트워크라면 게이트웨이의 MAC 주소를 2계층 헤더에 담습니다.
    3. **반복:** 데이터가 라우터를 거칠 때마다 3계층의 목적지 IP는 변하지 않지만, **2계층의 MAC 주소는 다음 장비의 주소로 계속 갈아 끼워지며** 이동합니다.
    
    > 3계층(IP)은 "어디로 가야 하는가"를 정하고, 2계층(MAC)은 그 과정에서 "이번에는 누구에게 전달해야 하는가"를 실행합니다.
    >
